简介

最近在看deno源码，遇到deno中使用FlatBuffers来处理js与rust，c++交互过程中的序列化和反序列化，对于我来说是个新的工具所以打算翻源码了解一下它的工作原理及deno使用它的理由。
FlatBuffers是一款序列化开源库，由Google开发，有市面上各种主流语言都有接口支持，主要的优点在于反序列化时不需要parse整块数据即可直接访问。

FlatBuffers VS JSON

JSON作为我们日常开发中使用最广泛的数据交换格式，这里拿FlatBuffers与JSON做个对比。

  • 读写性能
    JSON在解析数据并转化为特定语言的数据结构时会消耗我们的时间，生成很多临时对象浪费内存，特别是需要解析的数据量很大的情况下，而且如果要读取其中的一个字段需要解析整个JSON才能实现，这也是FlatBuffers对于JSON的最大优点，一切都是二进制读写，读取任意数据都不需要耗费资源，此外二进制数据更节省空间。
  • 易用性
    FlatBuffers首先需要写一个Schema，然后在实际使用过程中会增加很多代码量，在团队中也会遇到推广困难的问题，相比较JSON的无缝接入显然是个缺点。
  • 可读性
    由于是二进制数据所以这点和JSON没有可比性
  • 前后兼容性
    FlatBuffers提供前后兼容和Option字段，日常业务开发难免会有增减字段的需求，使用FlatBuffers无需担心由于数据格式兼容问题而导致crash。

我们可以看到JSON虽然性能远不如FlatBuffers，但是由于它的可读性和易用性，如果项目中没有性能要求的情况的下，还是推荐使用JSON。而在一些实时应用中，比如游戏中，FlatBuffers是更优的选择，而且在开发过程中发现使用FlatBuffers可以避免很多由于type引起的常见bug，api的设计也变得更清晰合理。

源码解析

我们抽取deno中的部分Schema来做实例分析，Base表示deno中js与rust交互的最外层Schema，其中的根据msg的不同分发到不同的处理函数中，例子中的FetchReq代表fetch code的request。

下面这段代码作用是发送一个获取http://example.com/index.ts内容的请求, msg_generated是使用flatc编译出来的js文件,提供针对上述Schema的api。

new flatbuffers.Builder();创建的builder实例中存放的就是当前Schema中的各种数据，并提供各种方法来使读写变得简单高效。
首先构造函数分配初始内存，默认1m,并初始化space,vtable等常用变量。

const urlOffset_ = builder.createString('http://example.com/index.ts')调用createString，如果传入的参数是Uint8Array则直接赋值，如果是String则用charCodeAt decode成uint8，这里为了简便默认参数s为英文,删掉了其他字符的处理代码。由于字符串以null结尾所以要先addInt8(0)，这里要说明的一点是FlatBuffers用的是小端模式，即从高内存地址往低内存地址写入。startVector中的prep有两个作用：对齐和内存不足申请额外内存。然后就是把utf8写入bytes中，最后endVector还要再写入当前字符串的长度占用4字节。所以长度27的字符串http://example.com/index.ts对齐后占用4的倍数28字节再加上4字节的长度，总共32字节，返回的urlOffset_就是在当前分配内存中的偏移量32。

所以如果我们传入了一个空字符串的话在flatbuffers中会占用多少空间呢？答案是1字节的null+3字节的对齐+4字节的长度=8字节。

回到一开始的js代码中startFetchReq作用是把字段的数量写入vtable中再赋值object_start。vtable的作用就是记录各字段的实际offset，这里fetchReq有2个字段url和id，类似于c中指针的作用吧。

注意：上面代码先createString再startFetchReq，如果我们换一下这两个函数的调用顺序会发生什么呢？不知道urlOffset_的情况下调用startObject中就无法知道当前实际的offset，所以flatbuffers规定createString必须在start***之前的原因就在这里。

endFetchReq最终会调用endObject，但js的实现没有注释实在是看不懂，所以看的是python版本的WriteVtable，vtable分成3个部分: vtable的大小，object的大小，N个字段的偏移量，这里N是2. 搜索vtables如果找到有现成的vtable则直接指向它，如果没有则新建一个vtable存下来，所以在object中的vtable offset可正可负，剩下的细节看注释即可。

返回fetchReqOffset后就开始创建Base table了，总体的思路和创建fetchReq table类似，在创建具体数据之后都是加上各种offset，最后来到finish.
